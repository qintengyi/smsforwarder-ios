## SmsForwarder 面板连接故障修复

### 问题
- Web 面板（smsf.xiaoyyua.top）显示设备"红米K40游戏增强版"无法连接
- 手机 SmsForwarder 模块（Magisk）已启用且正常运行

### 根本原因
**面板服务器默认网关路由问题**：服务器默认网关 192.168.1.12 无法路由到公网 frps（47.106.203.46），需改走小米路由器 192.168.1.1。

排查过程中还发现：
- 手机 daemon 监听 0.0.0.0:5000，但从面板服务器直连手机 HTTP 超时（TCP 可达但 HTTP 无响应，疑路由问题）
- 手机内嵌 frp v0.57.0，与 frps v0.61.1 不兼容
- SSH 地址 192.168.1.2 是网关转发设备，服务器真实 IP 是 192.168.1.8（ens2）和 192.168.1.9（ens6）

### 最终修复方案
1. 在面板服务器添加永久路由：`47.106.203.46 via 192.168.1.1 dev ens2`
   - systemd 服务：`route-frps.service`（开机自启）
2. 恢复手机 frpc 配置指向公网 `47.106.203.46:7105`
3. 数据库设备1 api_base_url 恢复为 `http://47.106.203.46:24000`
4. 停用临时本地 frps（已 disable）

### 验证结果（公网 frp 方案）
- 设备健康检查：online=true, 171ms ✅
- 配置查询：返回完整手机配置 ✅
- 电量查询：level=47%, Charging, Good ✅
- api_logs 全部 200 成功

### 关键架构信息
- 面板是 Go 编译二进制（smsforwarder-panel），非 Python Flask
- 部署路径：`/www/wwwroot/smsf.xiaoyyua.top/smsforwarder-panel-deploy/`
- 数据库：MySQL（smsf_xiaoyyua_top），表：users/devices/api_keys/api_logs/login_tokens
- 面板端口：12123，Nginx 反代 HTTPS
- frps 在公网 47.106.203.46:7105，Lucky 面板管理（端口 16601）
- 手机 daemon（Magisk 模块）内嵌 frp v0.57.0，配置 /data/adb/smsforwarder/frpc.toml
- 服务器双网卡：ens2(192.168.1.8) + ens6(192.168.1.9)，默认网关 192.168.1.12
- 小米路由器网关：192.168.1.1（手机和修复后的服务器到 47.106.203.46 走此网关）

### 黑边修复（第五轮，v1.3 build 5）— 找到根本原因
- **根本原因**：XcodeGen 的 `info` 键在 Xcode 26 中未正确设置 `INFOPLIST_FILE` 构建设置

- **修复**：移除 project.yml 的 `info` 键，改为显式设置 `INFOPLIST_FILE: Sources/SmsForwarder/Info.plist`
- 验证：编译后 app 内 Info.plist 正确包含 `UILaunchStoryboardName=LaunchScreen`、`UIRequiresFullScreen=true`、版本 1.3
- `LaunchScreen.storyboardc/` 正确编译并包含在 app bundle 中
- commit `4486de4`，GitHub Actions 编译成功（43 秒），ipa 494KB
- **教训**：XcodeGen 的 `info` 键在 Xcode 26 中不可靠，应显式设置 `INFOPLIST_FILE` 构建设置

### Turnstile 人机验证修复（v1.3 build 6）
- **根本原因**：WKWebView 的 baseURL 设为 `challenges.cloudflare.com`，但 site_key 注册域名是 `smsf.xiaoyyua.top`，域名不匹配导致验证失败
- **修复**：baseURL 改为面板服务器地址；使用显式渲染（`?onload=`回调）；添加超时检测和重试按钮

- **备用方案**：服务器端设置 `TURNSTILE_ENABLED=false` 可跳过人机验证

### 仪表盘数据解析失败修复（v1.3 build 7）
- **现象**：登录成功后仪表盘提示"数据解析失败:The data couldn't be read because it isn't in the correct format"
- **根本原因**：
  1. `DeviceConfig` 模型缺少灵活解码（字符串/数字混用导致 DecodingError）
  2. `DashboardViewModel.checkDeviceStatus()` 用 `async let` 并行调用，任一失败即全部失败
  3. `checkProxyHealth()` 走标准 `proxyCall`，设备 /health 返回非标准格式时解码失败
- **修复**：
  1. `DeviceConfig` 添加自定义 `init(from decoder:)`，用 `decodeFlexibleInt/Int64/Double` 容忍类型混用
  2. `checkDeviceStatus()` 改为 `async let` + `try?` 各自独立捕获错误，全部失败才报错
  3. `checkProxyHealth()` 重写为高度容错：直接发请求不走 proxyCall，兼容标准 APIResponse / 简单 JSON / 任意 200 响应
- commit `b65bd78`，GitHub Actions 编译成功（40 秒），ipa 500KB
