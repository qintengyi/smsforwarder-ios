## 灵动岛 WebSocket 重连风暴修复

**问题**：验证码发到手机后能手动查询，但灵动岛不触发。

**根因排查**：
- 服务器日志显示 iOS App 每 1-2 秒重连一次 WebSocket（重连风暴）
- 连接刚建立就被下一次 `connect()` 的 `teardown()` 取消，`subscribeAll()` 来不及执行
- Hub 轮询推送的短信根本无法到达客户端

**三重 bug**：
1. 重连级联：`connect()` → `teardown()` 取消旧 task → 旧 receive 回调 `.failure` → `reconnect()` → `connect()` → 无限循环
2. Timer 在后台线程不工作：`reconnect()` 后台线程调 `connect()` → `startPing()` 的 Timer 挂在后台 RunLoop（不运行）→ ping 不触发 → 150s 后被服务端断开
3. 无并发保护：多个 `reconnect()` 可同时触发

**修复**（WebSocketClient.swift）：
- 引入 `generation` 代次标记，旧连接的回调不再触发重连
- 所有状态操作切回主线程（`DispatchQueue.main.async`）
- `start/stop/connect` 全部在主线程执行
- `subscribeAll()` 连接后立即调用（不再延迟1秒）
- 添加 `print` 诊断日志（WSClient/LiveActivity/Monitor）

**编译坑**：`NSLog` 可变参数在 Swift 中不可用，改用 `print` + 字符串插值

**验证**：Nginx WS 反代配置正常（`$connection_upgrade` map 已定义，proxy_read_timeout 600s），后端 WS handler 正常（直连 127.0.0.1:12123 返回 401）

## 修复 Koishi QQ 机器人注册 sms_binding 唯一约束 Bug

### 问题
- QQ 机器人发送"注册sms"报 `ER_DUP_ENTRY: Duplicate entry 'onebot' for key 'sms_binding.unique:platform'`
- 第二次注册提示"用户名已存在"（users 表已有记录），但"登录sms"提示"你还未绑定面板账号"（sms_binding 表无记录）
- 根因：Koishi 插件 `koishi-plugin-smsforwarder` 的 `model.extend` 中 `unique: ['qq', 'platform']` 被 Koishi 解释为两个独立唯一索引（qq 唯一 + platform 唯一），而非开发者期望的复合唯一索引。platform='onebot' 全局唯一导致只有第一个用户能注册

### 架构
- `sms_binding` 表在 `koishidb` 数据库（非 `smsf_xiaoyyua_top`），由 Koishi 插件管理
- Koishi 插件源码：`/www/wwwroot/koishi/external/koishi-plugin-smsforwarder/`（src/index.ts + lib/index.js）
- 注册流程：Koishi 插件先调 Go 面板 `POST /api/bot/register` 创建 users 表记录 → 再写 koishidb.sms_binding 记录（两步无事务，第二步失败导致脏数据）
- Koishi 服务：systemd `koishi.service`，Node.js v24.12.0，yarn start

### 修复
1. 源码：`unique: ['qq', 'platform']` → `unique: [['qq', 'platform']]`（Koishi 复合唯一索引语法）
2. 数据库：DROP `unique:platform` + `unique:qq` 旧索引，保留 Koishi 重启时自动创建的 `unique:qq+platform (qq, platform)` 复合索引
3. 补数据：为 liye（QQ 3699288698, panelUserId=3）手动 INSERT sms_binding 记录
4. 重启 Koishi：`systemctl restart koishi`
5. 验证：`POST /api/bot/login-token {"qq":"3699288698"}` 返回 code=200 + login_url ✅

### Koishi 唯一索引语法
- `unique: ['a', 'b']` → 两个独立唯一索引 UNIQUE(a), UNIQUE(b)
- `unique: [['a', 'b']]` → 一个复合唯一索引 UNIQUE(a, b)

## 修复前后台切换 WebSocket 连接中断错误

**问题**：App 从后台恢复到前台时出现 `Software caused connection abort` 错误。

**根因**：iOS 进入后台时会强制中断 WebSocket 连接。之前 `start()/stop()` 用 `DispatchQueue.main.async` 异步执行，scenePhase 回调返回后 `teardown()` 才延迟执行，此时 iOS 可能已先中断连接，导致 receive 回调收到 `.failure` 并设置 `lastError` 暴露给用户。

**修复**（3 个文件）：
1. `WebSocketClient.swift`：
   - `start()/stop()` 在主线程时**同步执行**（`Thread.isMainThread` 判断），避免异步延迟
   - `receiveLoop` failure 分支检查 `shouldRun`——主动断开时不设置 `lastError` 也不触发 `reconnect()`
2. `MonitoringCoordinator.swift`：
   - 新增 `onBackground()`：设 `isBackground=true`，调 `ws.stop()` 主动断开
   - 新增 `onForeground()`：重置 `isBackground`，若 enabled+已登录则 `ws.start()` 重连
3. `SmsForwarderApp.swift`：
   - 添加 `@Environment(\.scenePhase)` + `.onChange(of: scenePhase)` 监听
   - `.background` → `onBackground()`，`.active` → `onForeground()`

**IPA 编译**：commit `3046ffb`，GitHub Actions run #29391458814，1m47s 成功

## 后台改用 HTTP 轮询替代 WebSocket（最终方案）

**问题**：即使有 audio 保活，`URLSessionWebSocketTask` 在后台仍被 iOS 强制中断（用户实测"还是会断"）。

**根因**：iOS 对 WebSocket 长连接有特殊后台限制，即使 App 进程未被挂起，WebSocket 仍可能被系统中断。这不是 App 层面能解决的问题。

**最终方案**：前台/后台双模式
- **前台**：WebSocket 实时推送（低延迟）
- **后台**：HTTP 轮询（每 5s 查 `/api/device/:id/proxy/sms/query`，按 date 去重）
- audio 保活贯穿前后台让 App 不被挂起，HTTP 请求在后台比 WebSocket 长连接可靠得多

**新增文件**：
- `Network/BackgroundPoller.swift`：后台轮询器
  - `Task` + `Task.sleep` 实现 5s 间隔轮询循环
  - `withTaskGroup` 并发轮询所有订阅设备
  - `lastMaxDate: [Int: Int64]` 按设备 ID 记录已知最大时间戳
  - 首次轮询建基线（不触发通知），后续 date > prevMax 的新短信触发 `onSMS` 回调
  - 响应解析兼容 `data:[]` / `data.list:[]` / `list:[]` 三种格式
  - date 字段兼容 NSNumber / Int / String

**修改文件**（commit `7c1a3cd`，v1.3 build 9）：
- `MonitoringCoordinator.swift`：
  - `onBackground()`：停 WS → 启 Poller
  - `onForeground()`：停 Poller → 启 WS
  - `apply()`：根据 isBackground 状态选择启动 WS 或 Poller
  - `setup()`：WS 和 Poller 共用 onSMS 回调路由到 LiveActivityManager
- `KeepAliveManager.swift`：音频中断恢复（来电后自动恢复）
- `SettingsView.swift`：调试面板显示轮询状态 + 上次轮询时间

   - `volume` 从 `0` 改为 `0.01`（非零，iOS 不检测静音）
   - 文件名从 `smsf_silence.wav` 改为 `smsf_keepalive_v2.wav`（强制重新生成）
   - 添加 `prepareToPlay()` + 详细错误日志（不再 `try?` 静默吞错）
   - 30s 健康检查 Timer：播放停止时自动 `play()` 重启
   - 监听 `routeChangeNotification`：耳机拔出后自动恢复播放
   - 暴露 `isKeepingAlive` / `lastError` / `startedAt` 供调试面板查看
2. `Info.plist`：`UIRequiresFullScreen` 从 `true` 改为 `false`（减少后台挂起概率）
3. `BackgroundPoller`：添加 `pollCount` 计数
4. `SettingsView`：显示音频保活状态/错误/启动时间/轮询次数

## 后台灵动岛不弹出 - 预启动+update方案（关键修复）

**问题**：日志显示"启动灵动岛"但灵动岛实际没弹出来。

**根因**：Apple 官方限制 `Activity.request()` 只能在 App 前台调用，后台调用会静默失败（错误只 print 不到 lastDebugLog，用户看不到）。即使 audio 保活让 App 在后台运行，`Activity.request()` 在后台仍然不工作。但 `activity.update()` 更新已存在的 Live Activity 在后台是允许的。

**方案**：前台预启动"待命"Live Activity → 后台收到验证码用 `activity.update()` 更新内容

**修改**（commit `2f3f799`，v1.3 build 12）：
1. `ActivityAttributes.swift`：`ContentState` 新增 `isIdle: Bool` 字段
2. `SmsForwarderWidget.swift`：
   - `isIdle=true`：灵动岛显示盾牌图标 + "验证码监听中"
   - `isIdle=false`：显示验证码（原逻辑）
3. `LiveActivityManager.swift` 全面重写：
   - `startStandby()`：前台预启动待命 Live Activity（isIdle=true）
   - `updateWithCode()`：有待命活动时用 `activity.update()`（后台可用），否则 `request()`（仅前台）
   - `scheduleResetToIdle()`：超时后 `resetToIdle()` 回到待命而非结束
   - `stopStandby()`：结束待命活动
   - 错误记录到 `lastDebugLog` 方便排查
4. `MonitoringCoordinator.swift`：
   - `apply()` 前台时调 `la.startStandby()`
   - `onForeground()` 确保待命存在（可能被系统清理后需重建）

**核心机制**：前台 `request()` 创建 → 后台 `update()` 更新（绕过 iOS 限制）

## 用 Activity.activities 查询真实状态 + pending 后备（commit `933e8da`，v1.3 build 13）

**问题**：build 12 的 `standbyActivity` 引用可能失效（系统清理活动后引用还在但活动已 `.ended`），`activity.update()` 静默失败，灵动岛不弹出。

**修复**：
1. 不再依赖 `standbyActivity` 引用，改用 `Activity<CodeActivityAttributes>.activities.filter { $0.activityState == .active }` 查询系统真实活动状态
2. 只更新 `.active` 状态的活动

4. `startStandby()` 发现已有活跃活动时复用而非重复创建
5. 调试面板新增：`standbyActive`（待命活动是否存在）、`lastUpdatePath`（update/request/pending）、`pendingCount`（待显示数量）

- 灵动岛日志显示 "⚠️无活动且request失败" → 确认 Activity.request 在后台被拒绝
