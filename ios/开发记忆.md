## 二、Go 面板 API 架构

### 2.1 API 逆向分析

通过分析面板前端 JS（`/assets/index-C10jmACY.js`）提取完整 API 结构：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/auth/turnstile` | GET | 获取 Turnstile 配置 `{enabled, site_key}` |
| `/api/auth/login` | POST | 登录认证 `{username, password, turnstile_token}` → `{token, id, username, remark}` |
| `/api/auth/profile` | GET | 获取用户信息 |
| `/api/devices` | GET | 设备列表 `[{id, name, api_base_url, sign_key, remark}]` |
| `/api/device/:id/health` | GET | 面板级设备健康 `{online, duration_ms}` |
| `/api/device/:id/proxy{path}` | POST | **设备代理** — 转发所有数据请求到设备 |

### 2.2 设备代理端点

所有设备数据请求通过 `POST /api/device/:id/proxy{path}` 转发：

| 代理路径 | 参数 | 返回 |
|----------|------|------|
| `/health` | `{}` | 设备在线状态 |
| `/config/query` | `{}` | 设备配置信息 |
| `/battery/query` | `{}` | 电量信息 |
| `/location/query` | `{}` | 定位信息 |
| `/sms/query` | `{type, page_num, page_size, keyword}` | 短信列表 |
| `/sms/send` | `{sim_slot, phone_numbers, msg_content}` | 发送结果 |
| `/call/query` | `{type, phone_number, page_num, page_size}` | 通话列表 |
| `/contact/query` | `{phone_number, name}` | 联系人列表 |
| `/contact/add` | `{phone_number, name}` | 添加结果 |
| `/wol/send` | `{mac, ip, port}` | 唤醒结果 |

### 2.3 统一响应格式

```json
{
  "code": 200,
  "msg": "success",
  "data": { ... }
}
```

- `code == 200` 表示成功
- `code == 401` 表示未认证（需重新登录）
- 列表接口的 `data` 可能是数组 `[...]`，也可能是 `{list: [...]}`

## 三、iOS 代码架构

### 3.1 网络层 (SmsForwarderAPI.swift)

**核心方法：**

- `buildAPIURL(path:)` — 构建面板 API URL（如 `/api/auth/login`）
- `buildProxyURL(deviceId:path:)` — 构建设备代理 URL（如 `/api/device/1/proxy/sms/query`）
- `get<T>(path:)` — 面板 API GET 请求
- `post<T>(path:body:)` — 面板 API POST 请求
- `proxyCall<T>(path:body:)` — 设备代理调用（自动附加 deviceId 和 token）
- `proxyFetchList<T>(path:body:)` — 代理调用并兼容 data 为数组或 `{list:[...]}`

**关键设计：**
- 所有请求自动附加 `Authorization: Bearer {token}` header
- `proxyCall` 超时 30 秒，面板 API 超时 15 秒，健康检查超时 15 秒
- 401 响应抛出 `authRequired` 错误，触发重新登录

### 3.2 灵活 JSON 解码

设备返回的 JSON 数据类型不固定（数字字段可能返回字符串如 `"85%"`、`"1700000000000"`），因此所有模型使用自定义 `init(from decoder:)` 配合灵活解码扩展：

```swift
extension KeyedDecodingContainer {
    func decodeFlexibleInt(forKey key: Key) -> Int?      // Int → Double → String("85%" → 85)
    func decodeFlexibleInt64(forKey key: Key) -> Int64?  // Int64 → Int → Double → String
    func decodeFlexibleDouble(forKey key: Key) -> Double? // Double → Int → String
}
```

**使用灵活解码的模型：** DeviceConfig, BatteryInfo, LocationInfo, SmsRecord, CallRecord, DeviceHealth

### 3.3 状态管理

- **SettingsStore** — 持久化 `serverURL` + `token` + `username` + `currentDeviceId` 到 UserDefaults
- **DeviceStore** (@Observable) — 管理设备列表和当前选中设备，持久化 currentDeviceId
- **AppStateManager** (@Observable) — 管理登录状态，集成 DeviceStore

### 3.4 Turnstile 人机验证

- 登录前调用 `GET /api/auth/turnstile` 检查是否启用
- 启用时用 `TurnstileView`（WKWebView）渲染 Cloudflare Turnstile
- 获取 token 后随登录请求一起提交
- **关键**：WKWebView 的 `baseURL` 必须设为面板服务器地址（与 site_key 注册域名匹配）
- 使用显式渲染（`?onload=onTurnstileLoad` 回调）替代自动渲染
- 10 秒超时检测 + 重试按钮 + WKNavigationDelegate 错误处理

### 3.5 Dashboard 错误隔离

仪表盘并行获取健康/配置/电量/定位数据，各请求独立捕获错误（类似 `Promise.allSettled`）：

```swift
async let healthTask = try? api.checkProxyHealth()
async let configTask = try? api.queryConfig()
async let batteryTask = try? api.queryBattery()

let healthResult = await healthTask    // Bool? — 失败返回 nil
let configResult = await configTask    // DeviceConfig? — 失败保留旧值
let batteryResult = await batteryTask  // BatteryInfo? — 失败保留旧值

isOnline = healthResult ?? false
if let cfg = configResult { config = cfg }
if let bat = batteryResult { battery = bat }

// 只有全部失败才报错
if healthResult == nil && configResult == nil && batteryResult == nil {
    errorMessage = "无法连接设备，请检查网络连接或设备是否在线。"
    showError = true
}
```

### 3.6 checkProxyHealth 高度容错

设备 `/health` 端点返回格式可能多样，因此 `checkProxyHealth()` 不走标准 `proxyCall`，直接发请求并多格式解析：

1. 网络错误（超时/无法连接）→ 返回 `false`
2. 尝试标准 `APIResponse<HealthData>` 格式 → 检查 `data.online` 或 `data.status`
3. 尝试简单 JSON `{online: true}` 或 `{status: "ok"}`
4. 有 code 字段但非标准 data → 检查 code==200
5. HTTP 200 但无法解析 → 保守返回 `true`（设备可达）

## 四、Bug 修复记录

### 4.1 iPhone 16 Pro 黑边问题（5 轮修复）

| 轮次 | 方案 | 结果 |
|------|------|------|
| 第 1-3 轮 | `UILaunchScreen` + `UIColorName` | 未解决 |
| 第 4 轮 | `UILaunchStoryboardName` + `LaunchScreen.storyboard` | 未解决 |
| **第 5 轮（最终）** | **移除 XcodeGen `info` 键，显式设置 `INFOPLIST_FILE`** | **解决** |

**根本原因：** XcodeGen 的 `info` 键在 Xcode 26 中未正确设置 `INFOPLIST_FILE` 构建设置，导致构建系统忽略了自定义 Info.plist，生成了默认 Info.plist（无 `UILaunchStoryboardName`、版本 1.0）。

**最终修复（project.yml）：**
```yaml
settings:
  base:
    GENERATE_INFOPLIST_FILE: "NO"
    INFOPLIST_FILE: Sources/SmsForwarder/Info.plist
    # 不要用 info: 键！
```

**Info.plist 关键配置：**
```xml
<key>UILaunchStoryboardName</key>
<string>LaunchScreen</string>
<key>UIRequiresFullScreen</key>
<true/>
```

**教训：** XcodeGen 的 `info` 键在 Xcode 26 中不可靠，应显式设置 `INFOPLIST_FILE` 构建设置。

### 4.2 Turnstile 人机验证加载失败

**现象：** 登录页显示"验证组件加载失败"

**根本原因：** WKWebView 的 `baseURL` 设为 `challenges.cloudflare.com`，但 Turnstile site_key 注册的域名是 `smsf.xiaoyyua.top`，域名不匹配导致 Cloudflare 拒绝渲染。

**修复：**
- `baseURL` 改为面板服务器地址 `https://smsf.xiaoyyua.top`
- 使用显式渲染（`?onload=onTurnstileLoad` 回调）替代自动渲染
- 添加 10 秒超时检测、重试按钮、WKNavigationDelegate 错误处理

**HTML 模板核心：**
```html
<script src="https://challenges.cloudflare.com/turnstile/v0/api.js?onload=onTurnstileLoad" async defer></script>
<script>
function onTurnstileLoad() {
    turnstile.render('#ts-container', {
        sitekey: 'SITE_KEY',
        callback: function(token) {
            window.webkit.messageHandlers.turnstile.postMessage({type: 'token', token: token});
        },
        'error-callback': function(code) {
            window.webkit.messageHandlers.turnstile.postMessage({type: 'error', message: '验证失败: ' + code});
        }
    });
}
</script>
```

### 4.3 仪表盘数据解析失败

**现象：** 登录成功后仪表盘提示"数据解析失败:The data couldn't be read because it isn't in the correct format"

**根本原因（三个）：**
1. `DeviceConfig` 模型缺少灵活解码 — 字符串/数字混用导致 `DecodingError`
2. `DashboardViewModel.checkDeviceStatus()` 用 `async let` 并行调用，任一失败即全部失败
3. `checkProxyHealth()` 走标准 `proxyCall`，设备 `/health` 返回非标准格式时解码失败

**修复：**
1. `DeviceConfig` 添加自定义 `init(from decoder:)`，用 `decodeFlexibleInt/Int64/Double` 容忍类型混用
2. `checkDeviceStatus()` 改为 `async let` + `try?` 各自独立捕获错误
3. `checkProxyHealth()` 重写为高度容错（见 3.6 节）

### 4.4 Git 推送问题

**现象：** `git push` 卡死，Windows GCM OAuth token 过期后非交互环境无法刷新。

**解决方法：** 用 PAT + 禁用凭据助手推送：
```bash
GIT_TERMINAL_PROMPT=0 git -c credential.helper= push https://x-access-token:{PAT}@github.com/qintengyi/smsforwarder-ios.git main
```

用 PAT 登录 `gh` CLI：
```bash
echo {PAT} | gh auth login --with-token
```

### 5.1 面板服务

### 5.2 网络拓扑

```
iPhone (iOS App)
    ↓ HTTPS
Nginx (smsf.xiaoyyua.top:443)
    ↓ proxy_pass
Go 面板 (127.0.0.1:12123)
    ↓ POST /api/device/:id/proxy{path}
frps (公网 47.106.203.46:7105)
    ↓ frp 隧道
手机 frpc (内嵌 daemon, Magisk 模块)
    ↓ 127.0.0.1:5000
SmsForwarder daemon (0.0.0.0:5000, sign_key=qty8520123)
```

### 5.3 服务器网络

| 项目 | 说明 |
|------|------|
| SSH | `ssh -p 5321 root@192.168.1.2`（密码 qty8520123），192.168.1.2 是网关转发 |
| 真实 IP | 192.168.1.8 (ens2)、192.168.1.9 (ens6) |
| 默认网关 | 192.168.1.12（部分公网 IP 不可达） |
| 永久路由 | `route-frps.service` 将 47.106.203.46 走 192.168.1.1 小米路由器 |

### 5.4 frp 配置

| 项目 | 说明 |
|------|------|
| frps 位置 | 公网 47.106.203.46:7105，Lucky 面板管理（端口 16601） |
| 手机 frpc | 内嵌在 smsforwarder-daemon（Magisk 模块），配置 `/data/adb/smsforwarder/frpc.toml` |
| frp 版本 | 手机内嵌 v0.57.0，frps 必须匹配（v0.61.1 不兼容） |
| 手机 daemon | 监听 0.0.0.0:5000，sign_key=qty8520123 |

### 5.5 面板连接故障修复

**问题：** Web 面板显示设备无法连接。

**根本原因：** 面板服务器默认网关 192.168.1.12 无法路由到公网 frps（47.106.203.46），需改走小米路由器 192.168.1.1。

**修复：**
1. 添加永久路由：`47.106.203.46 via 192.168.1.1 dev ens2`（systemd 服务 `route-frps.service`）
2. 恢复手机 frpc 配置指向公网 `47.106.203.46:7105`
3. 数据库设备1 `api_base_url` 恢复为 `http://47.106.203.46:24000`

## 八、技术要点速查

1. **XcodeGen + Xcode 26**：不要用 `info` 键，显式设置 `INFOPLIST_FILE` + `GENERATE_INFOPLIST_FILE: NO`
2. **Turnstile WKWebView**：`baseURL` 必须与 site_key 注册域名匹配
3. **JSON 灵活解码**：设备返回的数据类型不固定，用 `decodeFlexibleInt/Int64/Double` 兼容
4. **Dashboard 错误隔离**：`async let` + `try?` 各自独立捕获，类似 `Promise.allSettled`
5. **checkProxyHealth 容错**：不走标准 proxyCall，直接发请求多格式解析
6. **frp 版本匹配**：手机内嵌 v0.57.0，frps 必须匹配
7. **服务器路由**：47.106.203.46 需走 192.168.1.1 小米路由器网关
8. **Info.plist 启动屏**：用 `UILaunchStoryboardName`（非 `UILaunchScreen`+`UIColorName`）
9. **部署目标**：iOS 17（@Observable 宏需要）；Swift 5.9
10. **未签名构建**：`CODE_SIGNING_ALLOWED=NO`，.app → `Payload/` → zip 成 ipa
